# open-apply — 按 Spec 执行编码

Spec 是合同。遇到意外不硬来，提案更新。

## ⚡ 技能激活确认（必须执行）

你的第一条回复**必须完全输出**以下内容（将 `<变更名>` 替换为实际值）：

```
[open-apply v2.0] 按 tasks.md 执行编码 — Diff Preview → 逐 Task → 偏差提案 → Review Gate
[加载] 变更: <变更名> · spec.md: ✅/❌ · 分支: <分支名> · master/main: ✅/❌
[预览] 文件数: X 新增 / Y 修改 / Z 删除
```

输出此确认后，继续执行后续步骤。

## 前置检查

1. 找到 `ai-work/changes/<变更名>/` 目录
2. 确认 spec.md 已通过 HARD-GATE
3. 确认无 `[待澄清]` 项
4. 检查当前 Git 分支 — 禁止在 master/main 上编码

## Step 0: 输出「变更预览」（Diff Preview）

在执行任何编码前，输出一份完整的**变更影响声明**。用户确认后再开始：

```
========================================
  变更预览 — <变更名>
========================================

### 将修改的文件
| 文件 | 操作 | 原因 |
|------|------|------|
| UserController.java | ✏️ 修改 | 新增 /api/v2/users 入口 |
| UserService.java | ✏️ 修改 | 新增 findByIdV2，兼容旧方法 |
| UserDao.java | ✏️ 修改 | 新增 selectByIdV2 |
| UserDTO.java | ➕ 新增 | 新版用户响应对象 |
| UserMapper.xml | ➕ 新增 | 新 SQL 查询 |

### 不会修改的文件（防止 AI 过度修改）
- UserCache.java — 未涉及
- application.yml — 使用已有配置

### 外部影响
- 新增接口 /api/v2/users → 前端需适配
- 旧接口 /api/v1/users 保持不变 → 不影响现有调用方

========================================
用户确认后开始执行
========================================
```

**用户确认后**，进入 Task 执行模式。

## 执行模式

- **逐步**（默认）: 完成一个 task → 报告 → 等确认
- **批量**: 用户说"全部完成"→ 按顺序执行所有
- **紧急停车**: 逻辑冲突或 spec 缺失 → 立即停止

## 每个 Task 的步骤

1. 报告: "开始执行 Task-N: {描述}"
2. 按 spec 实现代码
3. **Verification 铁律**: 展示编译/测试输出
4. 在 tasks.md 中标记完成
5. 检查知识发现 → 写入 log.md
6. 更新 log.md 的「变更指标（Metrics）」章节中的编码阶段数据
7. Git commit: `[<变更名>] <中文简述>`
8. 报告完成

## 偏差处理（替代原"零偏差停车"）

遇到与 spec 不一致的情况时，**不是直接停车**，而是执行以下流程：

### 偏差检测条件
- 需要修改 spec 中未列出的文件
- 实际实现方式与设计不一致
- 发现技术障碍（依赖不兼容、API 废弃等）
- 需要引入未预期的依赖

### 偏差处理流程

```
[偏差检测] 检测到: <具体描述> (涉及文件: path/to/File.ext)
[原因] <技术原因分析，如"该 API 已在 v3 废弃，改用 v4 API">
[提案] 
  - 方案 A（推荐）: <修改方案，附带对 spec 的更新>
  - 方案 B（保守）: <尽量贴近原 spec 的方案>
  - 方案 C（放弃）: <停止此 task，回退>
[影响] 
  - 更新的 spec 章节: §6 数据变更 / §7 接口变更
  - 新增文件: xxx
  - 修改文件: xxx
[等待用户确认]
```

### 确认后的操作

| 用户选择 | 操作 |
|---------|------|
| "用 A" | 更新 spec.md/tasks.md → 继续执行 |
| "用 B" | 按方案 B 继续 |
| "放弃" | 回退此 task 的代码变更 → 标记 task 为 blocked |
| "修改后再试" | 用户编辑 spec → 重新确认 |

### 同步更新文档

偏差被接受后，必须同步更新：
1. **spec.md** — 更新相关章节（§ 数据变更 / § 接口变更 / § 配置变更 等）
2. **tasks.md** — 更新任务描述、涉及文件、验证命令
3. **log.md** — 记录决策

## Verification 铁律

每个 task 完成后**必须展示可验证的证据**。
**禁止**: "应该没问题"、"代码看起来正确"等无证据声明。
验证不通过时不得继续下一个任务。

## 完成

所有 task 完毕后:
1. 填写 tasks.md 底部的"完成"部分
2. 输出执行摘要（含实际 vs spec 偏差记录）
3. 提示: 执行 `skill open-review <变更名>` 进入审查
