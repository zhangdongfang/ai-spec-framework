# open-archive — 归档 + 知识沉淀

## ⚡ 技能激活确认（必须执行）

你的第一条回复**必须输出**以下单行确认：

```
[open-archive v1.1 | <变更名> | knowledge:✅/❌ | log:✅/❌] 开始归档
```

输出此确认后，继续执行后续步骤。

## 执行步骤

### Step 1: 知识提取

#### 知识分类（优先沉淀以下类型）

1. **踩坑型**：预期 vs 实际不一致（如"xxx 库 v3 不支持 yyy"、"某 API 返回格式与文档不同"）
2. **模式型**：发现可复用的代码模式（如"本项目处理分页的统一方式"、"错误处理的标准写法"）
3. **约束型**：未文档化的业务约束（如"状态从 A 到 B 必须经过 C"、"金额字段单位是分不是元"）
4. **性能型**：性能陷阱（如"批量查询超过 1000 条需分批"、"某接口响应 > 2s 时需降级"）

逐条展示 log.md 中的知识发现和踩坑记录，按上述分类标注类型，询问用户是否沉淀到 ai-work/knowledge/。

### Step 2: 写入知识文件

确认的知识写入 `ai-work/knowledge/<slug>.md`，格式：

```markdown
---
tags: [踩坑/模式/约束/性能]
created: YYYY-MM-DD
expires: YYYY-MM-DD（默认创建后 1 年，约束型不过期）
---
知识内容
```

更新 ai-work/knowledge/index.md:
```
- **触发关键词**: 一句话核心逻辑 -> `包名.类名.方法名` [标签] (过期: YYYY-MM-DD)
```

> **知识过期规则**：踩坑型/性能型默认 1 年过期，模式型默认 2 年过期，约束型不过期。过期知识在 index.md 中标注 `[已过期]`，不删除但降低检索权重。

### Step 3: 归档变更

将 `ai-work/changes/<变更名>/` 整个目录移到 `ai-work/archives/`，更新 spec.md 状态为 done。

### Step 4: 输出归档报告

输出一份归档摘要，供后续查阅：

```
========================================
  归档报告 — <变更名>
========================================
复杂度: L?
项目类型: <type>
文件数: X 新增 / Y 修改 / Z 删除
代码行数: +X / -Y
耗时: X 天
知识沉淀: X 条 → ai-work/knowledge/<slug>.md
关键决策: X 个
遗留问题: X 个
```
