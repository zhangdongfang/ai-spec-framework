# 模型路由与成本优化指南

## 核心原则

**强模型做决策，快模型做执行。** 不同阶段对 AI 能力的要求不同，合理分配模型可以在保证质量的同时显著降低成本。

## 阶段-模型匹配表

| 工作阶段 | 推荐模型级别 | 原因 | 典型模型 |
|----------|-------------|------|----------|
| **Research（调研）** | 强模型 | 需要理解复杂代码库、识别架构模式、发现隐含依赖 | Claude Opus / GPT-4o |
| **Propose（设计）** | 强模型 | 架构决策、Delta Spec 编写、风险识别需要高层推理 | Claude Opus / GPT-4o |
| **Apply（执行）** | 快模型 | Spec 已明确定义任务和边界，执行层面不需要复杂推理 | Claude Sonnet / GPT-4o-mini |
| **Review（审查）** | 强模型 | Spec 合规检查、代码质量审查需要全局视角 | Claude Opus / GPT-4o |
| **Archive（归档）** | 快模型 | 文件移动、格式化、索引更新等机械性操作 | Claude Sonnet / GPT-4o-mini |

## 复杂度级别的成本策略

### L1 简单任务（1-2 个文件）

- 全程使用**快模型**即可
- 无需 Spec，直接执行
- 预估成本：$0.01-0.05 / 任务

### L2 中等任务（3-8 个文件）

- 简化 Spec 编写：**强模型**（一次调用生成 Spec）
- 代码执行：**快模型**
- 预估成本：$0.10-0.30 / 任务

### L3 复杂任务（8+ 个文件）

- Research + Propose：**强模型**（2-3 次调用）
- Apply：**快模型**（多次调用）
- Review：**强模型**（1 次调用）
- 预估成本：$0.50-2.00 / 任务

## 工具配置方法

### Claude CLI

Claude CLI 支持在会话中切换模型：

```bash
# 用 Opus 做调研和设计
claude --model opus "请根据项目宪法，对以下需求进行 Research 调研..."

# 用 Sonnet 做执行
claude --model sonnet "请按照 Spec 任务清单逐项执行..."
```

或在交互模式中使用 `/model` 命令切换。

### OpenCode CLI

在 `opencode.jsonc` 中配置默认模型，执行阶段可通过命令行参数覆盖。

### IDE 插件

- **Cursor**：在设置中配置不同场景的模型偏好
- **Copilot**：使用默认模型即可，IDE 补全场景下成本已经很低

## 成本优化技巧

### 1. 上下文窗口管理

- **避免一次性喂入整个代码库** — Research 阶段有针对性地扫描相关文件
- **Spec 作为压缩上下文** — Apply 阶段只需传入 Spec 和相关源文件，无需完整对话历史
- **分段执行** — 大任务拆分为多个小 Apply 阶段，每次只传入当前任务所需的上下文

### 2. 缓存复用

- **知识库（knowledge/）减少重复调研** — AI 常犯的错误和架构决策记录在案，减少每次从头分析的开销
- **Spec 模板减少生成成本** — 固定结构让 AI 只需填充内容，不需要每次推导格式
- **规则文件减少纠正成本** — 明确的 boundaries 减少 AI 犯错后的返工

### 3. 避免浪费

| 浪费场景 | 优化方式 |
|----------|----------|
| AI 不了解项目规范，生成不合规代码后返工 | 加载 constitution.md 和 boundaries.md |
| AI 在遗留项目中重复造轮子 | Research 阶段强制扫描现有实现 |
| AI 偏离 Spec 设计，做了不需要的事 | "Plan is Contract" 规则约束执行范围 |
| Spec 与代码脱节，Review 阶段大量返工 | Reverse Sync 机制保持一致性 |
| 每次对话重新解释项目背景 | .ai-spec/ 配置文件提供持久化上下文 |

## 团队成本监控

建议跟踪以下指标：

| 指标 | 计算方式 | 优化目标 |
|------|----------|----------|
| 每任务平均 Token 消耗 | 月度总 Token / 完成任务数 | 逐月下降 |
| 一次通过率 | 无需返工的任务 / 总任务 | > 70% |
| 返工成本占比 | 返工消耗 Token / 总 Token | < 20% |
| 强模型使用占比 | 强模型 Token / 总 Token | < 40% |

> 框架的核心价值之一：通过前置的 Spec 设计和 Research 调研，减少后续 Apply 阶段的返工和浪费，从而降低总成本。
