# L3 Spec 模板（复杂，8+ 文件或跨模块）

```
# 需求名称
> status: propose | apply | review | done
> created: YYYY-MM-DD
> complexity: L3 复杂
> project_type: <项目类型>
> code-map: ai-work/changes/<name>/code-map.md

## 1. 背景与目标
## 2. 代码现状（Research Findings）
   - 2.1 相关入口与链路（含行号）
   - 2.2 现有实现
   - 2.3 可复用的模式
   - 2.4 发现与风险（引用 code-map.md 的变更耦合、测试契约）
## 3. 功能点
> 标注优先级：🔴Must / 🟡Should / 🟢Could

## 4. 业务规则
## 5. 验收标准
> 格式：Given 前置条件，When 操作，Then 预期结果。
> 每个功能点至少覆盖：正常 + 异常 + 边界。
## 6. 数据变更（Delta）— ADDED / MODIFIED / REMOVED
   - 表/集合变更
   - 数据迁移方案
   - 回滚数据方案
## 7. 接口变更
   - API 变更列表
   - 兼容性说明
   - 调用方升级计划
## 8. 配置变更
   - 新增/修改/废弃配置项
   - 环境变量
   - Feature flag
## 9. 架构决策与约束
| 决策 | 选择 | 放弃的方案 | 原因 |
|------|------|-----------|------|

架构约束（本变更不能违反的规则）：
- （如"不引入跨模块循环依赖"、"不绕过接口层直接调用"）
- 依赖方向: （只允许上层→下层）

## 10. 影响范围
   - 本服务
   - 外部依赖
   - 不在范围内: （明确列出不做什么，防止范围蔓延）
   - 变更耦合（引用 code-map.md）
## 11. 非功能性需求
   - 性能/SLA 要求
   - 安全要求
## 12. 业务可观测性
   - 12.1 关键业务指标（KPI）— 基线值 vs 目标值
   - 12.2 新旧对比策略（feature flag / 双写 / shadow read / 日志区分）
   - 12.3 灰度放量计划（百分比阶梯、观测周期）
   - 12.4 监控看板与告警规则（新增/修改的面板、告警阈值）
   - 12.5 回滚判断依据（"什么指标亮红灯时必须回滚"）
## 13. 回滚方案与兼容性
   - 代码回滚
   - 数据回滚
   - 前向/后向兼容性矩阵
## 14. 废弃与移除（如适用）
   - 废弃功能列表
   - Deprecation notice 计划
   - 数据清理策略
## 15. 部署计划
   - 部署顺序
   - 协调服务列表
   - 灰度策略
## 16. 测试策略
   - 单元测试: （覆盖哪些功能点，新增/修改用例数）
   - 集成测试: （跨组件交互，依赖桩/真实服务）
   - 端到端测试: （关键用户路径）
   - 性能测试: （基线对比，目标 QPS/延迟）
   - 回归范围: （变更影响区域内必须通过的已有测试）
## 17. 风险与关注点
## 18. 待澄清
## 19. 确认记录（HARD-GATE）
```
