# L1 Spec 模板（简单，1-2 文件）

```
# 需求名称
> status: propose | apply | review | done
> created: YYYY-MM-DD
> complexity: L1 简单
> project_type: <项目类型>

## 1. 背景与目标
为什么做 + 做完后的效果

## 2. 变更上下文
- 变更涉及文件: （文件路径列表）
- 为什么是这几个文件: （简短说明为什么改动范围如此）

## 3. 功能点
- [ ] 功能 1：（输入 -> 处理 -> 输出）

## 4. 验收标准
> 格式：Given 前置条件，When 操作，Then 预期结果。必须覆盖正常 + 异常场景。

- [ ] AC-01（正常）：Given ___，When ___，Then ___
- [ ] AC-02（异常）：Given ___，When ___，Then ___

## 5. 影响范围
- 不影响: （明确声明不受影响的范围）
- 不在范围内: （明确不做什么，如"不支持多字段排序"、"不处理历史数据迁移"）
- 风险: （即使是"无风险"也需声明）

## 6. 测试验证
- 验证方式: （单元测试 / 手动验证 / 编译通过即可）
- 需要补充的测试: （新增/修改哪些测试用例）

## 7. 业务可观测性
- 如何验证新功能正常: （如"对比新旧逻辑的日志输出"、"新增的指标面板"）
- 回滚判断依据: （什么指标异常时应回滚）

## 8. 技术决策（可选）
| 决策 | 选择 | 原因 |
|------|------|------|

## 9. 确认记录（HARD-GATE）
- **确认时间**:
- **确认人**:
```
