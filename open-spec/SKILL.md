# open-spec — 创建变更提案

将用户需求转化为结构化变更文档。自动评估复杂度，根据项目类型选择对应模板。

## ⚡ 技能激活确认（必须执行）

你的第一条回复**必须完全输出**以下内容：

```
[open-spec v3.0] 创建变更提案 — Check → Assess → Research → Converge → Draft → Output → HARD-GATE
[状态] rules/ 已读取: ✅/❌ · changes/ 状态: 无未完成/有未完成
[检测] 项目类型: <Java / Python / Frontend / Go / Node.js / Rust / Other>
[检测] 涉及现有代码: ✅ → 建议先跑 open-understand / ❌ 新功能 / ⚡ 混合(既有旧改又有新增)
```

输出此确认后，继续执行后续步骤。

## 前置

## Check: 是否涉及现有代码？

在执行复杂度评估前，先判断需求的类型：

| 类型 | 处理方式 |
|------|---------|
| **全新功能，不涉及任何现有代码** | 跳过此步，直接进入 Assess |
| **在老功能上修改/追加** | 先推荐用户跑 `skill open-understand` 产出 `code-map.md`，再继续 |
| **混合（新功能 + 老功能改造）** | 推荐跑 open-understand 理解受影响的老功能部分 |
| **不确定** | 默认走老功能路径，跑 open-understand |

如果已有 `code-map.md`（如之前生成过），检查其是否仍然有效（文件未被大幅修改），可直接复用。

**用户确认后**，进入 Assess。

## Assess: 复杂度评估

根据以下标准评估复杂度：

| 级别 | 标准 | 模板 |
|------|------|------|
| L0 | 1 个文件，1 个功能点，纯加逻辑无 DB/API/配置变更 | `spec-l0.md` |
| L1 | 改 1-2 个文件，单功能点，无跨模块，无 DB 变更 | `spec-lite.md` |
| L2 | 改 3-8 个文件，中等功能，涉及 1-2 个模块（可能有简单 DB 变更） | `spec-standard.md` |
| L3 | 8+ 文件或跨模块，新依赖，DB 迁移，涉及多服务协调 | `spec-full.md` |

**向上提升一级的条件**（满足任意一条即升一级）：
- 涉及 **DB schema 变更**（新增/修改表、字段、索引）
- 需要 **数据迁移**（现有数据的转换脚本）
- 涉及 **外部 API 不兼容变更**（请求/响应格式变化）
- 需要 **协调多服务部署**（非单服务变更）
- 涉及 **安全/权限模型变更**

向用户报告评估结果和推荐模板，用户可手动覆盖（如"用 L2"）。

**L0 特殊流程** — 如果评为 L0，跳过以下步骤：
- 跳过 Check（无需 code-map.md）
- 跳过 Converge（1 个文件无需提问收敛）
- 直接进入 Draft → Output

## Research: 代码与需求调研（自由度: 中）

每个结论必须附代码出处（文件路径 + 行号）。

1. **如果存在 code-map.md**：先读取该文件作为研究起点，直接复用其中的调用链路、影响雷达、变更耦合分析、测试契约
2. 定位需求涉及的现有代码（补充 code-map.md 未覆盖的部分）
3. 理解数据模型、API、业务逻辑
4. 识别调用链和依赖（参考 code-map.md 的全链路追踪结果）
5. 搜索 knowledge/ 是否有相关知识可复用
6. 记录可复用模式和风险（参考 code-map.md 的"风险与模式"章节）
7. 参考 code-map.md 的"测试契约"章节，了解已有测试覆盖和边界
8. 参考 code-map.md 的"变更耦合分析"章节，识别同改依赖

输出调研摘要，引用 code-map.md 中的章节（如"详见 code-map.md §3 调用链路"），等用户确认后继续。

## Converge: 逐个提问收敛 + 架构师 CheckList（自由度: 高）

- 一次只问一个问题或一组紧密相关的问题
- 优先给 2-3 个选项 + 推荐
- YAGNI 裁剪: 主动识别"nice to have"建议延后

### 架构师 CheckList（在执行收敛时逐项检查，不需要问的跳过）

**数据层检查：**
- [ ] 是否涉及 DB schema 变更？变更可回滚吗？（加列可回滚，删列/改类型不可）
- [ ] 是否需要数据迁移脚本？迁移过程中旧代码是否兼容？
- [ ] 是否引入新的存储介质（Redis / ES / S3 等）？

**接口层检查：**
- [ ] 新增/修改的外部接口是否与旧版本兼容？
- [ ] 是否需要 API 版本号提升（v1→v2）？
- [ ] 下游调用方是否需要同步修改？

**部署与发布：**
- [ ] 是否需要 feature flag / 灰度开关？
- [ ] 是否需要与其他服务协调发布顺序？
- [ ] 回滚方案是什么？（代码回滚 + 数据回滚）

**非功能性需求：**
- [ ] 对性能有要求吗？预估 QPS / 数据量级？
- [ ] 是否已有性能 baseline 可对比？
- [ ] 是否有新的监控指标 / 日志结构变更 / 告警规则需求？

**业务可观测性：**
- [ ] 新功能的关键业务指标（KPI）是什么？（如成功率、转化率、响应时间）
- [ ] 如何对比旧行为和新行为？（feature flag / 双写 / 日志分类 / Dashboard）
- [ ] 灰度/放量计划？（逐步放量百分比、观测周期）
- [ ] 回滚后如何确认恢复了旧行为？

**安全：**
- [ ] 新接口需要鉴权吗？（登录态 / API Key / RBAC）
- [ ] 变更是否暴露了之前未暴露的数据？

**废弃/移除检查：**
- [ ] 如果涉及功能废弃，是否有 deprecation notice 计划？
- [ ] 废弃数据的清理策略？

## Draft: 分段生成 Spec（自由度: 低）

分三段，每段确认后再继续:

1. **代码现状 + 功能点 + 业务规则** → 用户确认
2. **变更范围（Delta）+ 架构决策 + 非功能性需求** → 用户确认
3. **风险 + 回滚方案 + 待澄清** → 用户确认

## Output: 生成完整文档

在 `changes/<变更名>/` 下创建三个文件。**根据 Step 0 检测到的项目类型适配模板措辞和示例**。

### 项目类型适配规则

| 项目类型 | 构建验证命令 | 入口层名 | 业务层名 | 数据层名 |
|---------|------------|---------|---------|---------|
| Java/Spring | `mvn compile` / `./gradlew compileJava` | Controller | Service | DAO / Repository |
| Python | `pytest` / `python -m pytest` | Router / View | Service | Model / Repository |
| Frontend | `npm run build` / `npm run typecheck` / `npm run lint` | Page / Component | Hook / Store | API Client |
| Go | `go build ./...` / `go vet ./...` | Handler | Service | Repository / Store |
| Node.js | `npm run build` / `npx tsc --noEmit` | Route / Controller | Service | Model / Repository |
| Rust | `cargo build` / `cargo check` | Handler | Service | Repository |

以下模板中所有「入口层」「业务层」「数据层」等表述，以及验证命令、文件后缀，均按上述规则替换。

## 约束

- 待澄清全部解决前不允许进入 open-apply
- 引用 Research 发现的现有代码，不凭空设计
- 涉及资金/状态变更/权限变更 → 高亮提醒
- 验证命令根据项目类型自动适配，不硬编码
- 如果涉及废弃/移除，必须明确遗留数据处理方案
