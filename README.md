# AI Spec Framework — AI 辅助编码工作流

## 一句话

把 AI 编码从"黑箱生成"变成**可理解 → 可规划 → 可审查 → 可追溯**的结构化流程。

## 工作流总览

```
          ┌── 新功能 ──┐
需求 ──→  ├── 老功能 ──┤──→ ├──→ open-spec ──→ open-apply ──→ open-review ──→ open-archive
          └── Bugfix ──┘     ↑                     ↑  ↖  ↑         ↓
                           open-setup            open-understand  修正循环   ai-work/knowledge/
                                                      │
                                                    code-map.md
```

### 各 Skill 职责

| Skill | 什么时候用 | 产出 | 一句话 |
|-------|-----------|------|--------|
| **open-setup** | 新项目首次使用 | `ai-work/rules/` + `ai-work/AGENTS.md` | 扫描项目结构和编码风格，初始化上下文 |
| **open-understand** | 改老功能前 | `code-map.md` | 骨架提取→语义摘要→调用链→依赖关系图，支持增量分析 |
| **open-spec** | 每次变更前 | `ai-work/changes/<name>/spec.md + tasks.md + log.md` | 把需求转成结构化变更文档 |
| **open-apply** | Spec 确认后 | 实际代码 | 按 task 逐步编码，有偏差自动提案不硬来 |
| **open-review** | 编码完成后 | 审查报告 | AI 幻觉检查（含语义校验）→ Spec 合规 → 代码质量 |
| **open-debug** | 出 Bug 时 | 根因分析 + 修复 | 系统化调试，根因是 spec 问题就回溯 |
| **open-archive** | Review 通过后 | 归档 + 知识沉淀 + 回顾分析 | 归档变更，提取知识到 ai-work/knowledge/ |

## 使用说明

### 快速开始

1. 将 `ai-spec-framework/` 目录复制到你的项目根目录
2. 在你的 AI 编码工具中注册这 7 个 skill（每个 skill 对应一个 `SKILL.md` 文件）
3. 运行 `skill open-setup` 完成项目初始化
4. 开始使用：提需求 → `open-spec` → `open-apply` → `open-review` → `open-archive`

### 你的需求是什么？→ 选路径

```
┌─────────────────────────────────────────────────────────────────┐
│  我要做什么？                                                    │
├─────────────────────────────────────────────────────────────────┤
```

### 路径 A: 新项目初始化

**适用场景**：项目第一次使用本框架，需要让 AI 了解项目结构和编码风格。

```bash
skill open-setup
```

**AI 会做什么**：
1. 自动检测项目类型（Java / Python / Frontend / Go / Node.js / Rust）
2. 扫描目录结构、依赖文件、编码风格
3. 扫描外部服务依赖拓扑（HTTP/RPC/MQ/DB 调用链）
4. 生成 3 个文件：
   - `ai-work/rules/project-rules.md` — 项目类型、层名、构建命令、目录约定、外部服务依赖
   - `ai-work/rules/domain-rules.md` — 业务领域术语和约束
   - `ai-work/AGENTS.md` — AI 行为配置

**完成标志**：

```
[open-setup 完成]
- project-rules.md: ✅
- domain-rules.md: ✅
- AGENTS.md: ✅
```

> **注意**：后续所有 skill 都依赖 `project-rules.md`，所以新项目必须先跑这一步。

---

### 路径 B: 新功能开发

**适用场景**：新增一个功能，不涉及对老代码的理解。

**完整流程**（共 4 步）：

#### Step 1 — 规划变更（open-spec）

```bash
skill open-spec "我要加一个用户积分功能"
```

AI 执行过程：

```
① Check     检测是否涉及老代码 → 新功能则跳过 code-map
② Assess    复杂度评估 → 选择级别（L0/L1/L2/L3）→ 加载对应模板
③ Research  搜索代码库，收集可复用的模式和约束（附出处）
④ Converge  向你提问收敛需求 → YAGNI 裁剪不需要的功能
⑤ Draft     按模板分段撰写 spec → 每段让你确认
⑥ Output    输出 spec.md + tasks.md + log.md
⑦ HARD-GATE 你确认后 spec 才生效 ← 这是"红线"，不确认不编码
```

**产出文件**（在 `ai-work/changes/<变更名>/` 下）：
- `spec.md` — 变更提案（背景、功能点、验收标准、影响范围、技术决策）
- `tasks.md` — 编码任务拆解（每个 task 一行，可勾选）
- `log.md` — 过程日志（Research 发现、提问记录、偏差记录）

> 完整示例见 `open-spec/examples/l1-example.md`

#### Step 2 — 编码执行（open-apply）

```bash
skill open-apply <变更名>
```

AI 执行过程：

```
① 前置检查    读取 spec.md + tasks.md → 检测分支状态
② Diff Preview 列出"要改什么 / 不改什么"→ 你确认后才动手
③ 逐 Task 执行 每完成一个 task → 展示验证证据（编译/测试）→ 你确认
④ 偏差处理    遇到意外 → 自动出 3 个方案 → 你选一个 → 更新 spec
```

**执行模式**：
- **逐步模式**（默认）：一个 task 完成 → 报告 → 等你确认 → 下一个
- **批量模式**（仅 L0/L1）：你说"全部完成" → 按顺序执行所有 task
  - 遇到偏差自动降级为逐步模式
- **断点恢复**：对话中断后重新运行 `open-apply`，自动检测 tasks.md 中的完成状态，从中断处继续

#### Step 3 — 审查（open-review）

```bash
skill open-review <变更名>
```

AI 执行 3 个阶段的审查：

```
Stage 0: AI 幻觉检查（9 项）
  🔴 必须通过: 幻影 import / 幻影 API / 幻影依赖
  🟡 建议通过: 幻影配置 / 幻影类型 / 过度实现 / 参数语义 / 复制残留
  🟢 辅助检查: 死代码

Stage 1: Spec 合规性 + 架构合规
  逐条对照功能点和验收标准
  架构合规：无反向依赖、无循环依赖、未违反 spec 中的架构约束
  测试覆盖：新增/修改代码有对应测试
  输出：追溯矩阵（AC → 代码位置 → 测试位置 → 状态）

Stage 2: 代码质量
  命名 / 错误处理 / 性能 / 安全 / 可维护性
```

如果发现问题 → 进入修正循环 → 修完后重新检查 → 直到通过。

> 完整示例见 `open-review/examples/stage0-example.md`

#### Step 4 — 归档（open-archive）

```bash
skill open-archive <变更名>
```

AI 执行过程：

```
① 知识提取   从 log.md 中识别 4 类知识：
             踩坑型 / 模式型 / 约束型 / 性能型
② 知识确认   逐条展示 → 你选择哪些沉淀到 ai-work/knowledge/
③ 归档       变更目录移到 ai-work/archives/ → spec 状态改为 done
④ 回顾报告   Spec 预估 vs 实际对比（文件数/复杂度/偏差数）+ 改进建议
```

### 路径 C: 老功能修改

**适用场景**：修改/追加已有功能，需要先理解老代码。

**比路径 B 多一步**：先跑 `open-understand`。

#### Step 0 — 理解老代码（open-understand）

```bash
skill open-understand "用户登录模块"
```

AI 执行 8 个步骤：

```
Step 1  快速扫描 + 增量判断（已有 code-map 时只分析变更文件）
Step 2  骨架提取 — 提取所有函数/类声明，建立"文件×函数"矩阵
Step 3  全链路追踪 — 从入口层逐层追踪到存储层，交叉验证骨架覆盖率
Step 4  依赖关系追踪 — 结构化输出上下游依赖 + 跨文件 Import
Step 5  变更耦合分析 — git log 分析"哪些文件经常一起改"
Step 6  相似功能发现 — 查找可复用的已有实现
Step 7  测试契约提取 — 已有测试覆盖了什么、没覆盖什么
Step 8  配置/常量标注 — 改这个功能要同步改什么配置
```

**产出文件**：`ai-work/changes/<变更名>/code-map.md`，包含 11 个章节：

```
 1. 模块地图          — 目录结构 + 职责标注
 2. 文件语义摘要      — 每个文件一行摘要 + 函数级明细（⚠️ 标注不在调用链上的函数）
 3. 架构概览          — Mermaid 组件关系图
 4. 完整调用链路      — 入口→业务→数据→存储，每步附文件+行号
 5. 关键文件清单      — 文件 + 角色 + 风险等级
 6. 数据模型          — 表结构 / TypeScript 类型 / 状态定义
 7. 变更耦合分析      — 经常一起改的文件
 8. 依赖关系图        — 3 张结构化表（下游/上游/Import）
 9. 配置与常量        — 相关配置项和魔法值
10. 测试契约          — 已有测试覆盖和缺口
11. 风险与模式        — 可复用模式 + 风险提示
附-1. 文件指纹        — 增量分析用，记录分析时间和 git 修改时间
附-2. Git 考古        — 文件历史修改者和关键 commit
```

> **增量分析**：第二次对同一模块运行 `open-understand` 时，AI 会比对文件指纹表，只重新分析有变更的文件，未变更的部分直接复用。

然后继续路径 B 的 Step 1-4（open-spec → open-apply → open-review → open-archive）。`open-spec` 的 Research 阶段会自动读取 `code-map.md` 作为输入。

### 路径 D: 问题调试

**适用场景**：线上/开发环境出现 Bug，需要系统化定位和修复。

```bash
skill open-debug "用户登录后偶现 500 错误"
```

AI 执行 4 个阶段：

```
Phase 1: 根因调查（自由度: 中）
  复现 → 定位代码 → 追踪调用链 → 收集证据 → 提出假设
  DO:  搜索代码、阅读文件、追踪调用链、附证据
  DON'T: 修改代码、跳过复现、编造错误信息

Phase 2: 模式分析（自由度: 高）
  搜索项目中类似场景 → 找到 2+ 个修复方案 → 对比优缺点
  DO:  灵活搜索、查 knowledge 库、列多个方案
  DON'T: 修改代码、跳过知识库搜索

Phase 3: 假设验证（自由度: 低）
  选定方案 → 验证（单元测试/手动测试）→ 确认根因
  DO:  只验证一个方案、展示验证证据
  DON'T: 同时验证多个方案、跳过验证

Phase 4: 实施修复（自由度: 零）
  只改确认方案列出的文件 → 展示编译/测试结果
  回归测试铁律：必须写一个能复现此 Bug 的测试用例（先 fail 再 pass）
  DO:  只修改确认方案中的文件、展示验证证据
  DON'T: 重构、优化、引入新依赖
```

**根因分类后的走向**：
- **编码错误** → 直接在 Phase 4 修复
- **Spec 缺陷** → 回到 `open-spec` 更新，再走 `open-apply`
- **外部依赖** → 记录到 knowledge，必要时升级/替换

> 完整示例见 `open-debug/examples/debug-example.md`

### 路径 E: 极简模式（L0）

**适用场景**：只改 1 个文件、改 1 个点的小活（如修复一个 NPE、改一个默认值）。

```bash
skill open-spec "修复用户名的 NPE"
```

```
→ 自动评为 L0 → 跳过 Check/Converge → 直接 Draft + Output
→ 3 个章节的极简 spec（背景+功能点+验收标准）
→ 确认后 → open-apply（可用批量模式一步完成）
→ L0 快速通道：apply + review 合并执行，编码完直接跑 Stage 0 幻觉检查
```

整个流程 5-10 分钟完成。

| 级别 | 改动规模 | 典型场景 | Spec 模板 | 执行模式 |

### 进阶用法

#### 断点恢复

对话中断后重新运行 `skill open-apply <变更名>`，AI 会自动检测 `tasks.md` 中已完成的 task，从中断处继续。

#### 增量代码理解

对同一模块第二次运行 `skill open-understand "模块名"`，AI 会比对 `code-map.md` 中的文件指纹表，只重新分析有变更的文件。

#### 偏差处理

编码过程中 AI 遇到 spec 未预料的情况时，不会停车，而是自动输出 3 个方案让你选择，选完后同步更新 `spec.md`。

#### 批量模式

对 L0/L1 级别的变更，你可以对 AI 说"全部完成"，AI 会按顺序执行所有 task。如果中途遇到偏差，自动降级为逐步模式。

#### 知识复用

`ai-work/knowledge/` 中沉淀的知识会在后续 `open-spec` 的 Research 阶段和 `open-debug` 的 Phase 2 中被自动检索。越用越聪明。

## 核心设计原则

### 1. Diff Preview — 编码前先声明意图

编码前先声明"我要改什么、不改什么"，用户确认后再动手。

### 2. 偏差自动提案 — 取代零偏差停车

AI 遇到意外时**不直接停，而是出 3 个方案**让用户选，选完后自动更新 spec。

### 3. AI 幻觉检查 — 先扫幻觉再查质量

审查第一步不看 spec，而是跑编译 + grep 扫 9 种 AI 典型幻觉（含参数语义校验和复制残留检测）。

### 4. Verification 铁律 — 证据驱动

每个 task 完成后必须展示可验证的证据（编译输出/测试结果），禁止"应该没问题"。按级别设定最低验证标准：L0 编译通过 → L3 全量测试 + 集成测试。

### 5. 代码地图 — 借鉴 Codeindex 的结构化理解

改老功能前先产出 code-map.md。借鉴 Codeindex 思路：骨架提取（声明优先扫描）→ 两级语义摘要（文件级+函数级）→ 结构化依赖关系图（上下游+跨文件 Import）。支持增量分析——文件未变更时复用已有结果。

### 6. GWT 验收标准 — BDD 格式确保可测试

所有验收标准使用 `Given___，When___，Then___` 格式，每个功能点至少覆盖正常路径 + 异常/边界路径。

### 7. 架构合规检查 — 守住依赖方向

审查阶段检查架构合规：无低层→高层的反向依赖、无跨模块循环依赖、变更未违反 spec 中的架构约束。

### 8. 追溯矩阵 — AC 到代码到测试的全链路

审查必须输出追溯矩阵：`AC编号 → 实现位置(文件:行号) → 测试位置(文件:行号) → 状态`，确保每条验收标准都有代码落地和测试覆盖。

### 9. 回归测试铁律 — Bug 不能修两次

每个 bugfix 必须附带一个能复现此 Bug 的回归测试（先 fail 再 pass）。没有测试的修复不算完成。

### 10. 知识生命周期 — 带过期时间的知识沉淀

每条知识带分类标签和过期时间：踩坑型/性能型 1 年过期，模式型 2 年，约束型永不过期。过期知识标注 `[已过期]` 降低权重，不删除。

### 11. DO/DON'T 行为边界 — 消除模糊自由度

每个标注了"自由度"的步骤都有 DO/DON'T 列表，告诉 AI 具体能做什么、不能做什么。

### 12. 批量安全 + 断点恢复

批量模式仅限 L0/L1，遇偏差自动降级。断点恢复——对话中断后重新执行 open-apply，自动从 tasks.md 检测完成状态继续。

### 13. Token 效率 — 按需加载不重复

模板和适配表不内联在 skill 中，按需加载外部文件。后续 skill 读取 `project-rules.md` 获取项目信息，避免 7 种语言模板在每个 skill 中重复。

### 14. 累计效能度量 — 跨变更趋势追踪

归档报告聚合历史数据：偏差率、Review 缺陷密度、知识复用次数、平均变更耗时，与近 5 次平均值对比，标注趋势（↑↓→）。

## 设计理念 — 为什么这样设计

本框架的每个设计决策都不是拍脑袋的产物，而是来自四个专业视角的最佳实践。

### 系统架构师视角：可控性

**核心诉求**：AI 生成的代码必须在架构约束内，不能为了实现功能而破坏系统结构。

| 设计决策 | 参考实践 | 解决什么问题 |
|---------|---------|------------|
| Diff Preview（编码前声明意图） | Infrastructure as Code 的 `terraform plan` | AI 经常"善意地"多改文件，先声明范围让人审批 |
| 架构合规检查（反向依赖/循环依赖） | ArchUnit / Dependency-Check | AI 不了解分层约束，容易写出 Controller→DAO 的直连代码 |
| 分层验证标准（L0→L3） | 测试金字塔（Test Pyramid） | 小改动不需要跑全量测试，大改动不能只跑编译 |
| 偏差自动提案（3 选 1） | ADR（Architecture Decision Records） | 把"AI 遇到意外就停车"改为"提供决策选项"，保持推进力 |
| 外部服务依赖拓扑 | DDD 上下文映射图 | 变更前必须知道"改这个会影响谁"，否则上线就炸 |

**希望达到的效果**：AI 编码不再是黑箱——每一步都在架构围栏内执行，偏离时有标准处理流程，而不是事后发现。

### 需求分析师视角：可追溯性

**核心诉求**：从需求到代码到测试，每一步都能反向追溯，不存在"不知道为什么这样写"的代码。

| 设计决策 | 参考实践 | 解决什么问题 |
|---------|---------|------------|
| GWT 验收标准格式 | BDD（Behavior-Driven Development） | 模糊的 AC 导致实现和预期不一致，GWT 强制结构化 |
| 追溯矩阵（AC→代码→测试） | RTM（Requirements Traceability Matrix） | 审查时能一眼看出哪条 AC 没实现、哪条没测试 |
| Research 阶段强制引用出处 | 引用溯源 | AI 容易编造不存在的 API/文件，强制附出处杜绝幻觉 |
| 业务规则清单（code-map §4.5） | 领域建模中的不变量提取 | 代码中隐含的业务规则不写下来就会在修改时被意外破坏 |
| MoSCoW 优先级标注（L3） | MoSCoW 优先级排序法 | 大需求中区分 Must/Should/Could，防止范围蔓延 |

**希望达到的效果**：每行代码都能回答"为什么写这个"——向上追溯到 AC，向下追溯到测试用例。

### 专业测试视角：可验证性

**核心诉求**：AI 写的代码必须有证据证明是对的，不能靠"代码看起来没问题"过关。

| 设计决策 | 参考实践 | 解决什么问题 |
|---------|---------|------------|
| Verification 铁律（每步展示证据） | CI/CD 门禁 | AI 经常声称"已完成"但实际编译都过不了 |
| 回归测试铁律（Bug 不能修两次） | TDD Red-Green-Refactor | 修了 Bug 不加测试 = 下次还会修同一个 Bug |
| AI 幻觉检查 9 项（含参数语义+复制残留） | 静态分析 + Code Review Checklist | AI 特有的错误模式（编造 API、复制后忘改）需要专项检查 |
| 测试策略章节（L2/L3 模板） | 测试计划 | 不能等写完才想"要不要写测试"，spec 阶段就规划测试范围 |
| 测试契约提取（code-map §10） | 契约测试 | 改老代码前先知道已有测试覆盖了什么、没覆盖什么 |

**希望达到的效果**：零信任——不信 AI 的自我报告，只信编译器输出、测试结果、grep 验证。

### 产品经理视角：可度量性

**核心诉求**：每次变更的效果和过程都可量化，团队能持续改进而不是凭感觉。

| 设计决策 | 参考实践 | 解决什么问题 |
|---------|---------|------------|
| 业务可观测性（KPI 基线 vs 目标） | DORA Metrics / SLO | "上线了但不知道跑得对不对"是最常见的产品风险 |
| 归档回顾分析（预估 vs 实际） | Sprint Retrospective | 不做回顾就无法改进 |
| 累计效能指标（偏差率/缺陷密度/知识复用） | Engineering Metrics Dashboard | 单次变更的数据没意义，跨变更趋势才有意义 |
| 知识生命周期（分类+过期） | 知识管理（Knowledge Management） | 知识不是越多越好，过时的知识比没有知识更危险 |
| 渐变复杂度（L0→L3） | 精益思维（Lean） | 不同规模的变更需要不同重量的流程，否则小改动也被流程拖死 |

**希望达到的效果**：团队能回答"我们的 AI 编码效率是在变好还是变差"——用数据说话，不用感觉。

## 评价

### 优势

| # | 优势 | 说明 |
|---|------|------|
| 1 | **闭环完整** | 从项目初始化到归档，覆盖整个开发生命周期，没有断裂 |
| 2 | **多语言原生** | 7 种项目类型各有一套层名、术语、构建命令、外部服务依赖扫描 |
| 3 | **防幻觉机制** | Stage 0 九项检查 + Verification 铁律 + Diff Preview，三层防护 |
| 4 | **渐变复杂度** | L0 极简→L3 跨模块，L0 还有快速通道（apply+review 合并） |
| 5 | **知识飞轮** | 按分类沉淀知识，带过期时间管理，过时知识降权不删除 |
| 6 | **全链路可追溯** | GWT 验收标准 + 追溯矩阵（AC→代码→测试），审查时一目了然 |
| 7 | **架构守护** | 审查阶段自动检查反向依赖和循环依赖，spec 中声明架构约束 |
| 8 | **回归测试铁律** | 每个 bugfix 必须附回归测试，Bug 不能修两次 |
| 9 | **累计效能度量** | 归档时聚合历史数据，偏差率/缺陷密度/知识复用跨变更趋势追踪 |
| 10 | **偏差友好** | 遇到意外出 3 个方案而非停车，选完自动同步 spec |
| 11 | **断点恢复** | 对话中断后自动检测 tasks.md 状态从中断处继续 |
| 12 | **Token 高效** | 模板和适配表按需加载，不在每个 skill 中重复 |

### 劣势

| # | 劣势 | 说明 |
|---|------|------|
| 1 | **流程较重** | 即使 L0 也需要走 3 个 skill（spec→apply→review），习惯"直接改"的开发者会觉得繁琐 |
| 2 | **文档维护成本** | spec/tasks/log 等多份文档需要同步更新，自动同步能力有限 |
| 3 | **团队落地门槛** | 需要整个团队统一使用这套流程，新人需要学习 7 个 skill 的用途和顺序 |
| 4 | **没有 CI 集成** | 当前流程都是人工触发，没有自动化的 CI 门禁（比如自动跑 review） |
| 5 | **审查仍然依赖人** | Stage 0 和 1 可以自动跑，但最终代码质量审查还是需要人来看 |
| 6 | **跨语言一致性** | 虽然每个语言有对应模板，但对小众/混合项目（如 Flutter、.NET、Monorepo）覆盖不足 |

### 适合的场景

- **中等复杂度功能**（L1-L3）：改 2-20 个文件，需要多人 review 的变更
- **新成员加入**：不熟悉代码库，需要 code-map.md 快速上手
- **多人协作项目**：需要规范的变更记录和知识沉淀
- **高变更频率模块**：反复修改的老功能，code-map.md 可复用

### 不适合的场景

- **紧急修复**（线上 P0）：等不了流程，直接改代码
- **极度简单的项目**（如只有一个文件的脚本）：全套流程是负担
- **单人副业项目**：不需要这么多文档和审查
