# open-debug — 系统化调试

铁律：**禁止在未确认根因前直接改代码。**

## ⚡ 技能激活确认（必须执行）

你的第一条回复**必须完全输出**以下内容：

```
[open-debug v1.0] 系统化调试 — 根因调查 → 模式分析 → 假设验证 → 实施修复
[加载] 问题: <用户描述的问题>
```

输出此确认后，继续执行后续步骤。

## Phase 1: 根因调查（自由度: 中）

1. 复现问题：确认触发条件和错误现象
2. 定位代码：找到问题所在的文件、类、方法
3. 追踪调用链：从入口到问题点的完整路径
4. 收集证据：日志、堆栈、变量状态

输出：问题根因的初步假设，附带代码出处。

### 根因分类决策

定位根因后，判断问题类型：

| 根因类型 | 示例 | 后续路径 |
|---------|------|---------|
| **编码错误** | 空指针、逻辑写反、拼写错误 | 继续 Phase 2-4 修复 |
| **Spec 缺陷** | spec 遗漏了某个边界条件、验收标准不合理 | → 建议 `skill open-spec <变更名>` 更新 spec |
| **理解偏差** | open-understand 的 code-map 有误 | → 建议更新 code-map.md |
| **外部依赖** | 第三方库升级、API 不兼容 | → 建议更新 spec 的依赖章节后继续 |

如果根因是 spec 缺陷或理解偏差，**不在 debug 里直接改代码**，而是输出：

```
[根因分类] 检测到: Spec 缺陷 / 理解偏差
[原因] <具体分析>
[建议] 执行 `skill open-spec <变更名>` 更新 spec 后重新走 open-apply
[或] 手动编辑 spec.md 的 §<章节> 后继续当前 debug
```

## Phase 2: 模式分析（自由度: 高）

1. 对比项目中类似场景的处理方式
2. 搜索 knowledge/ 是否有相关踩坑记录
3. 分析是偶发问题还是系统性问题
4. 识别可能的修复方案（至少 2 个）

输出：修复方案对比，给出推荐。

## Phase 3: 假设验证（自由度: 低）

1. 选择推荐方案，说明为什么
2. 在本地验证方案可行性（如写最小复现）
3. 确认方案不会引入新问题

输出：确认的修复方案。

## Phase 4: 实施修复（自由度: 零）

1. 实施修复
2. Verification 铁律：展示验证结果（测试通过/编译通过/问题不再复现）
3. Git commit: `fix: <修复描述>`
4. 如有新发现，写入 `changes/<变更名>/log.md`（如果当前有活跃变更）或 `knowledge/<slug>.md`

## 约束

- 每个阶段必须输出明确结论才能进入下一阶段
- 如果 Phase 1 无法定位根因，扩大搜索范围，不要猜测
- 修复后必须验证问题不再复现
