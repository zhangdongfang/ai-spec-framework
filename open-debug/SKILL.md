# open-debug — 系统化调试

铁律：**禁止在未确认根因前直接改代码。**

## ⚡ 技能激活确认（必须执行）

你的第一条回复**必须输出**以下单行确认：

```
[open-debug v1.1 | 问题: <用户描述的问题摘要>] 开始系统化调试
```

输出此确认后，继续执行后续步骤。

## Phase 1: 根因调查（自由度: 中）

- **DO**: 搜索代码、阅读日志、追踪调用链、收集证据
- **DO**: 每个假设附代码出处（文件路径 + 行号）
- **DON'T**: 修改任何代码文件
- **DON'T**: 跳过复现步骤直接猜测根因
- **DON'T**: 编造不存在的错误信息或堆栈

1. 复现问题：确认触发条件和错误现象
2. 定位代码：找到问题所在的文件、类、方法
3. 追踪调用链：从入口到问题点的完整路径
4. 收集证据：日志、堆栈、变量状态

输出：问题根因的初步假设，附带代码出处。

> 完整推理链示例见 `open-debug/examples/debug-example.md`

### 根因分类决策

定位根因后，判断问题类型：

| 根因类型 | 示例 | 后续路径 |
|---------|------|---------|
| **编码错误** | 空指针、逻辑写反、拼写错误 | 继续 Phase 2-4 修复 |
| **Spec 缺陷** | spec 遗漏了某个边界条件、验收标准不合理 | → 建议 `skill open-spec <变更名>` 更新 spec |
| **理解偏差** | open-understand 的 code-map 有误 | → 建议更新 code-map.md |
| **外部依赖** | 第三方库升级、API 不兼容 | → 建议更新 spec 的依赖章节后继续 |

如果根因是 spec 缺陷或理解偏差，**不在 debug 里直接改代码**，而是输出：

```
[根因分类] 检测到: Spec 缺陷 / 理解偏差
[原因] <具体分析>
[建议] 执行 `skill open-spec <变更名>` 更新 spec 后重新走 open-apply
[或] 手动编辑 spec.md 的 §<章节> 后继续当前 debug
```

## Phase 2: 模式分析（自由度: 高）

- **DO**: 灵活搜索项目中的类似模式、搜索 knowledge/、对比多种方案
- **DO**: 识别至少 2 个修复方案并给出推荐
- **DON'T**: 修改任何代码文件
- **DON'T**: 跳过 ai-work/knowledge/ 的搜索

1. 对比项目中类似场景的处理方式
2. 搜索 ai-work/knowledge/ 是否有相关踩坑记录
3. 分析是偶发问题还是系统性问题
4. 识别可能的修复方案（至少 2 个）

输出：修复方案对比，给出推荐。

## Phase 3: 假设验证（自由度: 低）

- **DO**: 选择推荐方案并说明理由，本地验证可行性
- **DO**: 确认方案不会引入新问题
- **DON'T**: 同时验证多个方案（一次只验证一个）
- **DON'T**: 跳过验证直接进入修复

1. 选择推荐方案，说明为什么
2. 在本地验证方案可行性（如写最小复现）
3. 确认方案不会引入新问题

输出：确认的修复方案。

## Phase 4: 实施修复（自由度: 零）

- **DO**: 只修改 Phase 3 确认方案中列出的文件
- **DO**: 修复后必须展示验证结果（编译/测试/复现检查）
- **DO**: 有新发现写入 log.md 或 knowledge/
- **DON'T**: 重构周边代码、添加"顺便优化"的改动
- **DON'T**: 修改不在修复范围内的文件
- **DON'T**: 未经确认引入新依赖

1. 实施修复
2. **回归测试铁律**：为本次修复的 bug **必须添加一个能复现此 bug 的测试用例**（先写测试验证 fail，再修复验证 pass）。确保此 bug 不会再次出现
3. Verification 铁律：展示验证结果（新增回归测试通过 + 编译通过 + 问题不再复现）
4. Git commit: `fix: <修复描述>`
5. 如有新发现，写入 `ai-work/changes/<变更名>/log.md`（如果当前有活跃变更）或 `ai-work/knowledge/<slug>.md`

## 约束

- 每个阶段必须输出明确结论才能进入下一阶段
- 如果 Phase 1 无法定位根因，扩大搜索范围，不要猜测
- 修复后必须验证问题不再复现
- **Bug 不能修两次**：每个 bugfix 必须附带回归测试，没有测试的修复不算完成
