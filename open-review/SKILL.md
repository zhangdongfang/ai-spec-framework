# open-review — 审查 + AI 幻觉检查

不信报告，只信代码。先扫 AI 幻觉，再查质量。

## ⚡ 技能激活确认（必须执行）

你的第一条回复**必须输出**以下单行确认：

```
[open-review v2.1 | <变更名> | tasks:✅/❌ | rules:✅/❌] 开始审查
```

输出此确认后，继续执行后续步骤。

## 前置

1. 找到 `ai-work/changes/<变更名>/`
2. 确认 tasks.md 中所有 task 已完成
3. 读取 ai-work/rules/ 作为审查基线

## Stage 0: AI 幻觉检查

**在检查 spec 合规之前，先用编译器和静态分析扫一遍 AI 典型幻觉。**

读取 `ai-work/rules/project-rules.md` 获取构建命令和依赖声明文件（如 pom.xml / package.json / go.mod 等）。

| 幻觉类型 | 检查方式 | 通过标准 |
|---------|---------|---------|
| 🔴 **幻影 import** | 运行项目构建命令（见 project-rules.md） | 编译/转译 0 错误 |
| 🔴 **幻影 API** | grep 确认调用的方法/函数在目标文件中存在 | AI 引用的每个方法签名都能在对应文件中找到 |
| 🔴 **幻影依赖** | 检查依赖声明文件中是否有对应依赖 | 无未在构建文件中声明的 import |
| 🟡 **幻影配置** | grep 检查引用的配置项在配置文件中定义 | 代码中读取的每个配置 key 都有对应的默认值或定义 |
| 🟡 **幻影类型/接口** | 检查引用的自定义类型/接口确实被定义 | 类型引用确实存在 |
| 🟡 **过度实现** | 逐文件比对 spec 功能点 vs 实际代码 | 无 spec 未要求的方法/组件/路由 |
| 🟡 **参数语义** | 检查核心业务函数的调用参数是否与 spec 业务规则一致 | 关键路径的参数值匹配 spec 中的业务规则描述 |
| 🟡 **复制残留** | 检查是否存在从相似功能复制但未修改的魔法值/表名/路由/常量 | 无残留的旧模块标识符 |
| 🟢 **死代码/未使用变量** | 运行 lint 检查 | lint 无 warning 或有解释 |

### 输出格式

```
## AI 幻觉检查报告
### 🔴 必须修复
- [FAIL] 幻影 import: xxx.java 引用了 `com.example.NotExist`，该包不存在
### 🟡 建议修复
- [WARN] 幻影配置: UserService.java 引用了 `${user.new.field}`，但 application.yml 中未定义
### 🟢 通过
- [PASS] 编译通过 0 错误
- [PASS] 所有类型引用存在
### 结论
PASS / FAIL（存在 🔴 项即 FAIL）
```

> 完整 Stage 0 检查报告示例见 `open-review/examples/stage0-example.md`

**存在 🔴 幻觉 → 进入修正循环修复后再继续。**
**🟡 幻觉 → 视严重程度决定是否修复。**

## Stage 1: Spec 合规性 + 架构合规

逐条比对（直接检查源码，不看 AI 摘要）：

### 审查清单
- [ ] 功能点覆盖：每个功能点标注实现位置（文件 + 方法名）
- [ ] 验收标准满足：逐条 AC 验证
- [ ] Delta 变更落地：ADDED/MODIFIED/REMOVED 全部到位
- [ ] YAGNI 检查：无过度实现、未引入未提及的依赖
- [ ] 架构合规：无低层→高层的反向依赖、无跨模块循环依赖、变更未违反 spec 中的架构约束
- [ ] 测试覆盖增量：新增/修改的代码有对应测试，测试覆盖率不低于变更前

### 追溯矩阵（必须输出）

```
| AC 编号 | 实现位置（文件:行号） | 测试位置（文件:行号） | 状态 |
|---------|---------------------|---------------------|------|
| AC-01   | xxx.java:45         | xxxTest.java:23     | ✅   |
| AC-02   | xxx.java:78         | （无测试）           | ⚠️   |
```

**FAIL（任何不合规）→ 进入修正循环**

### 输出格式
```
## Spec 合规审查报告
### 合规项
- [PASS] 功能点 1：实现在 xxx.java:123
### 不合规项
- [FAIL] AC-02：实现不完整
### 架构违规
- [FAIL/WARN] xxx.java 中 Controller 直接调用了 DAO，绕过 Service 层
### 测试覆盖
- [WARN] 新增 3 个方法，仅 1 个有测试覆盖
### YAGNI 违规
- [WARN] 引入了未提及的 xxx 依赖
### 结论
PASS / FAIL
```

## Stage 2: 代码质量

按严重程度分级（直接检查源码）：

| 级别 | 说明 | 处理 |
|------|------|------|
| Critical | 安全漏洞/数据损坏风险 | 必须修复，阻塞合并 |
| Important | 编码规范违反/潜在 Bug | 应修复 |
| Minor | 代码风格/命名建议 | 建议修复 |

### 审查维度
1. **编码规范** — 命名/日志/异常处理（参考 `ai-work/rules/`）
2. **安全红线** — 硬编码密钥? 日志泄露敏感信息? 输入未校验?
3. **代码质量** — 重复代码? 死代码? 过长方法? 错误处理不完整?
4. **业务安全** — 资金/状态/权限变更是否显式校验?

**存在 Critical → 进入修正循环**

## 修正循环

发现问题时:

1. 列出所有问题，按 Critical → Important → Minor 排序
2. 逐条修复，每条修复后:
   - 展示验证结果（编译/测试输出）
   - 同步更新 spec.md + tasks.md + log.md（文档同步铁律）
   - Git commit: `[<变更名>] fix: <修正描述>`
3. 修复完成后重新执行 Stage 0 → Stage 1 → Stage 2

用户可说"先不修 Minor"跳过建议级问题。

## 测试建议

审查完成后，基于变更内容给出测试建议:

- 哪些路径需要补充单测
- 边界条件和异常场景
- 如用户要求"写测试"，按 Red/Green TDD 执行

## 最终报告

- 全部 PASS → 提交 PR 进入人工审查，提示 `skill open-archive <变更名>`
- 存在问题 → 修正后重新审查
